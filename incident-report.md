# Отчёт об инциденте: подозрительные SSH-подключения

## 1. Краткое описание

В рамках учебной SOC-лаборатории был выполнен анализ SSH-событий на Linux-системе. В журналах были обнаружены попытки подключения под несуществующим пользователем, а также соединения, закрытые на этапе предварительной аутентификации.

Данные события были созданы в контролируемой лабораторной среде для отработки базового процесса SOC L1: сбор логов, поиск подозрительных событий, анализ, классификация и подготовка рекомендаций.

## 2. Среда

**Система:** Linux VM
**Сервис:** OpenSSH Server
**Источник логов:** `journalctl -u ssh`
**Инструменты:** `journalctl`, `grep`, `ss`, терминал Linux

## 3. Обнаруженная активность

В логах были обнаружены следующие типы событий:

* попытка входа под несуществующим пользователем;
* закрытие соединения до успешной аутентификации;
* SSH-события с пометкой `[preauth]`.

Пример события:

```text
Invalid user INVALID_USER from LOCALHOST_IPV6 port 40312
Connection closed by invalid user INVALID_USER LOCALHOST_IPV6 port 40312 [preauth]
```

## 4. Анализ

Событие `Invalid user` означает, что на SSH-сервер была выполнена попытка входа под пользователем, которого нет в системе. В реальной инфраструктуре такие события могут быть признаком подбора имён пользователей, автоматического сканирования или подготовки к перебору паролей.

Пометка `[preauth]` означает, что соединение было закрыто до прохождения аутентификации. В данном случае успешного несанкционированного входа обнаружено не было.

Источник `LOCALHOST_IPV6` указывает на то, что события были сгенерированы локально в рамках учебной лаборатории.

## 5. Оценка критичности

**Уровень критичности:** средний.

Сами по себе такие события не подтверждают компрометацию системы. Однако при большом количестве похожих попыток, особенно с внешних IP-адресов, они могут указывать на подозрительную активность и требуют дополнительной проверки.

## 6. Рекомендации

Рекомендуемые меры защиты:

1. Использовать вход по SSH-ключам.
2. Отключить парольную аутентификацию, если она не требуется.
3. Запретить вход под пользователем root по SSH.
4. Ограничить SSH-доступ с помощью firewall.
5. Регулярно анализировать SSH-логи.
6. Проверять пользователей с sudo-правами.
7. При необходимости использовать Fail2Ban или аналогичные средства защиты.
8. Отслеживать успешные SSH-входы после серии неуспешных попыток.

## 7. Вывод

В ходе лабораторной работы был отработан базовый SOC-процесс: просмотр логов SSH-сервиса, поиск подозрительных событий, анализ попыток входа, описание правил обнаружения и оформление отчёта об инциденте.

Успешного несанкционированного входа обнаружено не было.
