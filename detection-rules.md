# Правила обнаружения

## Правило 1. Попытка входа по SSH под несуществующим пользователем

**Описание:**
Правило выявляет попытки подключения по SSH под пользователем, которого нет в системе.

**Источник логов:**
`journalctl -u ssh` или `/var/log/auth.log`

**Логика обнаружения:**

```bash
sudo journalctl -u ssh --since "today" | grep -Ei "invalid user"
```

**Пример события:**

```text
Invalid user INVALID_USER from LOCALHOST_IPV6 port 40312
```

**Уровень важности:**
Средний.

**Почему это важно:**
Попытки входа под несуществующими пользователями могут указывать на подбор учётных записей, автоматическое сканирование или подготовку к brute-force-атаке.

**Рекомендуемые действия:**

* проверить источник подключения;
* посмотреть количество похожих событий;
* проверить, были ли успешные входы после этих попыток;
* ограничить SSH-доступ при необходимости;
* использовать вход по SSH-ключам;
* отключить вход под root по SSH.

---

## Правило 2. Закрытие SSH-соединения на этапе предварительной аутентификации

**Описание:**
Правило выявляет SSH-соединения, которые были закрыты до успешной аутентификации.

**Источник логов:**
`journalctl -u ssh` или `/var/log/auth.log`

**Логика обнаружения:**

```bash
sudo journalctl -u ssh --since "today" | grep -Ei "connection closed|preauth"
```

**Пример события:**

```text
Connection closed by invalid user INVALID_USER LOCALHOST_IPV6 port 40312 [preauth]
```

**Уровень важности:**
Низкий или средний.

**Почему это важно:**
Само по себе закрытие соединения до аутентификации не означает взлом. Однако большое количество таких событий может быть признаком сканирования, перебора пользователей или попыток несанкционированного доступа.

**Рекомендуемые действия:**

* проверить частоту событий;
* проверить, повторяется ли один и тот же источник;
* проверить, есть ли попытки входа под разными пользователями;
* проверить наличие успешных SSH-входов после подозрительных событий;
* оставить парольный вход отключённым, если используется вход по ключам.
